# 開発ロードマップ

最終目標: Markdown に ` ```chord ` ブロックを追加し、数字ディグリー記法 DSL で書いたコード譜を HTML 表示できるブログプラットフォームを、markdown.mbt (mizchi/markdown) のフォークとして構築する。

本ロードマップは 2026-07-03 時点の両リポジトリの実地調査(実装・仕様書・統合ポイント・先行事例)に基づく。現状評価・アーキテクチャ判断・先行事例の詳細は [development-plan.md](./development-plan.md) を参照。

---

## 戦略: 垂直スライス

3つの戦略案(A: ライブラリ先行完成 / B: 垂直スライス / C: プロダクト逆算)を独立に立案し、「ソロ開発の実行可能性」と「プロダクト価値・アーキテクチャ健全性」の2つの観点で審査した結果、**案B: 垂直スライス戦略**を採用する(経緯は development-plan.md §10)。

核となる考え方:

1. **最短で端到端を通す** — 「Markdown 中の ` ```chord ` ブロックがブラウザでコード譜表示される」一本を Phase 2 で成立させる
2. **動く画面を判断基準にする** — 以降のパーサ書き直し・CSS 反復・仕様拡張はすべて、動くプレビューへの回帰として検証する
3. **応急修正と恒久化を分離する** — 最大の技術リスクであるパーサ全面書き直し(fail-fast 廃止)は、回帰検出手段(動くプレビュー+テスト)を得てから Phase 3 で行う

依存関係: 1 → 2 → 3 → 4 → 5 → 6 の直列が基本。ただし Phase 3(パーサ書き直し)と Phase 4 の CSS 反復は、Phase 2 で常設するスタンドアロンプレビューページ上で部分的に並行できる。

## Phase 3: 堅牢化 — エラー耐性・位置情報・仕様 v1.0 整合

**ゴール**: 動くスライスを土台に、パーサをエディタ用途(ライブプレビュー・エラー箇所表示)に耐える品質へ引き上げ、仕様書と実装の乖離をゼロにする。本ロードマップ最大の技術リスク箇所であり、Phase 2 の動くプレビューを回帰検出器として使う。

### タスク

1. **fail-fast の廃止**: 行単位でエラーを収集し部分 AST を返す設計へ変更(markdown.mbt の CST 的エラー耐性思想と、1 文字ごとの不完全入力に耐えるライブプレビューの要件に合わせる)
2. `ParseError.position` の実装と、`derive(ToJson)` + `@json` による構造化 JSON 出力(kind / line / position / token)。手組み JSON 文字列連結のエスケープ不備も同時解消
3. `lexer` / `chord_parser` の StringView / パターンマッチベース書き直し(Phase 1 応急修正の恒久化)
4. 仕様整合修正 → `docs/chord.md` 改訂(v1.1.x):
   - DuplicateSeventh 検出の実装(または仕様から削除)
   - InvalidQuality の発生条件定義
   - 仕様書内の正規表現 `sus[1-7]` → `sus[246]` 修正(本文・AST 定義と矛盾している)
   - コメント構文 `#` の正式定義(仕様書の構文例が既に使用しているのに未定義)
   - 括弧内空白の可否、空入力の扱いの明記
   - 将来予約トークンの明記: ChordPro grid 語彙(`||`, `|.`, `|:`, `:|`, ボルタ `|1`/`:|2`, `%` 等)
5. **フロントマターのパース実装**(仕様 v1.2): ブロック先頭の `---` 区切り、`key` フィールドの検証(`A`〜`G` + `#`/`b`)、`ScoreAST.meta` への格納、JSON 出力の `meta` 対応、エラー(UnclosedFrontmatter / InvalidFrontmatterField / InvalidKey)
6. テスト拡充(仕様の構文例・エラー一覧の未カバー分): `1M7b5` / `1sus4` 成功例、`1M7b5(b9)` 混在、`1/b3`、`17` vs `1M7` 優先順位、`16`/`36`、`dim7`/`aug`、`add2`/`add4`/`add6`、`1(,b9)` カンマ不正、括弧不一致、不正テンション、`1M77` 重複、空入力、フロントマター(正常系・エラー系)
7. **仕様とテストの対応表**の作成: `docs/chord.md` の成功例テーブル・エラー一覧の全行にテストケースを対応づける

### 完了条件

- [ ] 複数の不正行を含む入力で、全エラーが行・位置つきで返り、正しい行のコード譜は描画されることをプレビューで確認できる
- [ ] 仕様とテストの対応表が存在し、`docs/chord.md` の全構文例・全エラー例に対応するテストが全パスする
- [ ] エラー JSON に kind / line / position が含まれ、`JSON.parse` 可能なことをテストで検証済み(引用符・バックスラッシュ・制御文字を含む入力で確認)

## Phase 4: 表示品質と SSR — ブログ本番の描画

**ゴール**: プレビュー(JS 経路)に加え、MoonBit 側 `render_html`(SSR / 静的 HTML)経路で chord ブロックを描画し、CSS・キー表示・読者向け UI を含む表示品質を実用レベルにする。

### タスク

1. **SSR 配線**: markdown.mbt フォークの `renderer.mbt` `Block::FencedCode` 分岐に `info == "chord"` フックを追加するか、未配線の `RenderOptions.code_highlighter` を `render_html` に貫通させる(決定ポイント D6)
2. スコープ付き CSS 生成 API(ChordSheetJS の `cssString('.scope')` 方式)、flexbox 小節グリッドレイアウト、`@media print` 対応
3. ディグリー → 実音変換: AST + key → 実コード名の**純関数**として実装(パーサには手を入れない)。tonal.js の `Progression.fromRomanNumerals` / ChordSheetJS の `toChordSymbol(key)` を参照実装とする。♯/♭ 表記選択を含む
4. **タブ UI**: レンダリング済みブロックに「ディグリー」「コード」の 2 タブ(仕様 v1.2 の表示仕様)。コードタブの初期キーはフロントマターの `key`
5. **キープルダウン**: コードタブで任意のキーへ移調表示。再レンダリングはクライアントでも chord-language の JS ビルド(MoonBit 製・単一ソース)を呼ぶ方式を推奨(決定ポイント D13)
6. エディタ内シンタックスハイライト: `src/highlight_chord/` パッケージ新設(`highlight_common.tokens_to_html` 再利用)→ `frontend/highlight/languages/chord.ts` → `types.ts` の `highlightLanguages` / `languageAliases` → `package.json` の imports / exports / files の 3 箇所追記
7. **配布方式の決定**(実施は Phase 6): mooncakes.io publish / monorepo 化 / git submodule のいずれかをこのフェーズ末までに決める(決定ポイント D5)

### 完了条件

## Phase 5: DSL 仕様 v1.3 — 実用曲が書ける表現力

**ゴール**: 動くビューを見ながら DSL を拡張し、実在の楽曲 1 曲を丸ごと記述・表示できる表現力に到達する。1 機能ごとに「パーサ → レンダラ → プレビュー確認」の小スライスで反復する。

キー指定(フロントマター)は v1.2 として仕様策定済み・Phase 3〜4 で実装済みの想定。このフェーズでは構造の表現力を拡張する。

2. リピート記号・ボルタ(Phase 3 で予約した ChordPro grid 語彙の実装)
3. N.C.(ノーコード)/ 休符・終止線

## リスクと緩和策

| リスク | 緩和策 |
|--------|--------|
| HEAD 実装は一度もコンパイル・テスト実行されておらず、ロジックバグが潜在 | Phase 1 でテスト実行基盤を先に確立し、失敗を Phase 3 書き直しの入力として扱う。パッチ投資は最小限 |
| moonbitlang/core の API ドリフト再発(String 系のラベル引数化・StringView 化が進行中) | ツールチェーンバージョンの固定記録(Phase 1)、markdown.mbt フォークと同一ツールチェーンでの整合確認 |
| fail-fast 廃止はパーサ中核の再設計で、Phase 2 のレンダラに手戻りが出うる | Phase 2 レンダラは使い捨て前提と明示。ゴールデンテストを HTML 出力側に置き、AST 変更に耐える検証面を確保 |
| 小節線グリッドの Web CSS は先行 OSS 実例が薄く、自前設計の反復量が読めない | Phase 2 からスタンドアロンプレビューページを常設し、playground 統合と切り離して見た目を反復 |
| パス依存は CI・Cloudflare ビルドで壊れる | chord-language 単体 CI を Phase 1 で先行導入。配布方式の決定を Phase 4 末に前倒し(実施は Phase 6) |
| `ast-renderer.tsx`(codeBlockHandlers)は npm 未公開の playground 内部コードで、本家追従コストが恒常発生 | Phase 6 でフォーク差分ドキュメント+取り込み手順を整備し、差分を一覧管理 |
| 現行仕様の表現力不足(セクション・リピート欠如)で、Phase 4 時点では実用記事が書きにくい | Phase 3 で ChordPro grid 語彙を予約トークンとして先に仕様化し、v1.3 での破壊的変更を防ぐ |
| 1 人で DSL・パーサ・フォーク・フロントエンド・デプロイの全レイヤーを保守する負荷 | 各フェーズを「動く縦切りスライス」で閉じ、中断・再開可能な状態を常に保つ |
| ライセンス混在(Apache-2.0 / MIT) | Phase 3 までに NOTICE 方針を確認し、Phase 6 で表記を最終整理 |
