# コード譜言語仕様書

## 概要

このドキュメントは、数字ディグリー記譜法に基づくコード譜言語の仕様を定義します。

## 基本構造
```
入力 = [フロントマター] 行*
行   = コード行 | セクションラベル行 | 歌詞行
コード行 = (コード | グループ | 小節線 | 反復 | NC | 空拍)* [コメント] 改行
セクションラベル行 = "[" 名前 "]" [コメント] 改行
歌詞行 = ">" 断片* 改行
グループ = コード ("-" コード)+
反復 = "%"
NC = "NC" | "N.C."
空拍 = "_"
```

フロントマター(ブロック先頭でのキー宣言)は v1.2 で追加された仕様。詳細は「[Chord ブロックとフロントマター](#chord-ブロックとフロントマターv12)」を参照。

### パースの基本方針（v1.2.2）

- **fail-fast はしない**。エラーのある行は結果から除外してエラーとして収集し、正しい行のパースは継続する（部分 AST + エラー列を返す）。ライブプレビューで 1 文字ごとの不完全入力に耐えるための設計
- エラーには**行番号**（元ソース基準、1-indexed）と **position**（行内の 0-indexed 文字位置）が付く
- **空入力・空白のみの入力**は、行を持たない空の楽譜として正常に扱う（エラーではない）

## トークンの定義

### 1. コード (Chord)

空白文字（スペース、タブ等）で区切られた文字列で、以下の構造を持ちます：
```
コード = [アクシデンタル] ディグリー [クオリティ] [セブンス] [オルタレーション] [括弧内オプション] [スラッシュベース] [色アノテーション]
```

#### 構成要素

- **アクシデンタル**: `s` (シャープ) | `b` (フラット)
- **ディグリー**: `1` | `2` | `3` | `4` | `5` | `6` | `7`
- **クオリティ**: `m` (マイナー) | `dim` (ディミニッシュ) | `aug` (オーギュメント)
- **セブンス**: `7` | `M7` | `6` | `add` + Addディグリー
  - **パース優先順位**: `7`/`M7`/`6` が `add` より優先的にマッチ
  - **Addディグリー**: `2` | `4` | `6` | `9` | `11` | `13`
- **オルタレーション**: `b5` (フラットファイブ) | `sus` + ディグリー (サスペンド)
  - 括弧外のオルタレーション（後方互換性のため保持）
- **テンション**: `b9` | `9` | `s9` | `11` | `s11` | `b13` | `13`
- **括弧内オプション**: `(` + オプション + (`,` + オプション)* + `)`
  - **オプション**: テンション | オルタレーション
  - 空の括弧 `()` はエラー
  - カンマの不正使用（先頭・末尾・連続）はエラー
  - **括弧の内側では空白を許容する**（`1M7(b9, s11)` は 1 つのコードトークン。v1.2.2 で明文化）
- **スラッシュベース**: `/` + [アクシデンタル] + ディグリー
  - スラッシュがある場合、ディグリーは**必須**
  - アクシデンタルは**オプショナル**
- **色アノテーション**: `@` + 色名（`blue` | `red` | `green`）
  - コードトークンの**末尾**にのみ置ける（スラッシュベースよりも後）
  - ブログ記事中で文脈上目立たせたいコードに使う**表示用アノテーション**（音楽的な意味は持たない）
  - 色名は小文字のみ。`blue` / `red` / `green` 以外はエラー

#### グループ（v1.3 — `-` で 1 スロットを共有）

`-` でコードを連結すると、複数のコードが**表示上の 1 スロット**を共有する（四分音符に対する八分音符のような分割のイメージ）:

```
6m7 3m/5 4m 2-5    # 最後のスロットを 2 と 5 が等間隔で分け合う
1-2m7-5@red        # 3 連結。色は個別のコードに付く
```

- 各パートは通常のコードとして完全にパースされる（色アノテーションも個別に付けられる）
- 空のパート（`2-` / `-5` / `2--5`）はエラー。区切り文字だけのトークン（`---` 等）は 1 つの UnrecognizedToken になる
- 括弧の内側では分割しない
- `~`（v1.3.0 の旧記法・ChordPro grid 由来）も互換エイリアスとして動作する
- **設計判断**: `-` をグループ区切りに採用したため、ジャズ記法のマイナー略記（`6-` = 6m）は**将来もサポートしない**。マイナーは常に `m` で書く

### 2. 改行 (LineBreak)

コードとコードの間に改行文字 (`\n`) が入った場合、楽譜上の改行として扱われます。空行は無視されます（行番号のカウントには含まれる）。

### 3. 小節線 (BarLine)

`|` 文字は楽譜の小節区切りを表します。

### 3b. 反復記号 `%`（v1.3 — 実装済み）

`%` は**直前のコードの反復**（同じコードが続くこと）を表す省略記号です。独立したトークンで、空白なしでも区切られます（`1%` は `1` と `%`）。

```
| 1M7 % | 4 % |     # 1M7 1M7 | 4 4 と同じ意味
```

- パーサは反復の解決（どのコードを指すか）を行わず、AST には `Repeat` トークンとして残る。表示はディグリー・コード両タブとも `%`
- `%` に色アノテーション等は付けられない

### 3b-2. N.C.（v1.5 — 実装済み）

`NC` または `N.C.` と書くと「コードなし」を表すトークンになります（表示は常に `N.C.`）。ブレイクやストップタイムに使います。

```
| NC 1 | N.C. 4 |
```

- 1 スロットを占有し、**歌詞を割り当てられる**（ストップタイムで歌う場面）
- 色アノテーション等の注釈は付けられない（`NC@red` はエラー）
- グループ（`-` 連結）の中では使えない

### 3b-3. 空拍 `_`（v1.5 — 実装済み）

`_` は**空の拍**を表すトークンです。スロットを 1 つ占有しますが、コードは表示されません。拍位置を明示したいとき（コードを伸ばす・歌詞だけ置く）に使います。

```
| 1 _ _ _ |          # 1 を 4 拍分に伸ばすイメージ
> ひ か り の        # 空拍にも歌詞を割り当てられる
```

- 空白なしでも区切られる（`1_4` は `1` `_` `4` の 3 スロット）
- **歌詞の割り当て対象**（`||` 由来の空セルは対象外）
- グループの中では使えない

### 3c. セクションラベル（v1.4 — 実装済み）

`[名前]` を**単独行**で書くと、セクションラベルとして扱われます。中身はフリーテキストです（自動採番はしない）。

```
[Aメロ]
| 1M7 % | 4M7 5 |

[サビ]
| 4M7 5 | 3m7 6m7 |
```

- ラベル行にもコメント（`#` 以降）を書ける。そのためラベル名に `#` は使えない
- `[A] 1 4` のように同じ行に他のトークンがある場合はラベルではなく、通常のコード行として解釈される（`[A]` は認識できないトークンとしてエラー）
- 表示は**白抜きの反転バッジ**（背景 = 本文の文字色、文字 = ページ背景色。ライト/ダーク両テーマで自動反転）
- AST では `sections: [{line, name}]` として行番号つきで保持される

### 3d. 歌詞行（v1.4 — 実装済み）

コード行の**直後の行**に `>` で始まる行を書くと、空白区切りの断片が直前のコード行のコード（Chord / グループ / `%`）へ**左から順に**割り当てられます。表示は各コードの真下です。

```
| 1M7 % | 4M7 5 |
> ひかりの _ さきへ ゆこう

| 6m7 3m7 | 2-5 1 |
> きみと あるく この-みち へ
```

- **完全にオプショナル**: 歌詞行が無ければコード行の書き方・表示は一切変わらない
- `_` 単独 = そのコードには歌詞なし（スロットを 1 つスキップ）
- 断片中の `_` は表示上のスペースになる（`sweet_home` → `sweet home`。断片を分けずにスペースを含めるための記法）
- 断片がコード数より**少ない**のは正常（残りは歌詞なし）。**多い**場合は TooManyLyricFragments エラー（先頭から割り当てた上でコード行の描画は維持される）
- グループ（`2-5`）への断片は、`-` でグループのコード数どおりに分割できる場合に各コードへ配られる（`この-みち` → `2` に「この」、`5` に「みち」）。分割できない場合は先頭のコードにまとめて付く
- 歌詞行は**自由テキスト**であり、`#` もコメントにならず歌詞の一部になる
- 歌詞行の直前の行がコード行でない場合（ブロック先頭・空行やラベルの直後・歌詞行の連続）は OrphanLyricLine エラー
- コードタブ（実音表示）・任意キーへの移調でも歌詞は保持される

#### 予約トークン（将来拡張）

以下の小節線バリエーションは ChordPro grid 語彙を正として**将来のために予約**されています（v1.4 候補）。現行仕様での動作: `||` は 2 つの小節線、`%%` は 2 つの反復記号として解釈されます。将来のバージョンで意味が変わる可能性があるため、使用しないでください。

`||`（複縦線）、`|.`（終止線）、`|:` `:|` `:|:`（リピート）、`|1` `:|2`（ボルタ）、`%%`（2小節リピート）

### 4. コメント（v1.2.2 で正式定義）

`#` から行末まではコメントとして無視されます。行頭に置けば行全体がコメントになります。

```
1 4 5 | 1   # イントロ
# この行はすべてコメント
```

## 構文例

### 基本形
```
1          # ディグリー1
3m         # ディグリー3マイナー
b7         # フラット7
s4         # シャープ4
```

### セブンス付き
```
5M7        # ディグリー5メジャーセブンス
2m7        # ディグリー2マイナーセブンス
47         # ディグリー4ドミナントセブンス
36         # ディグリー3シックス
4add6      # ディグリー4 add6
4add9      # ディグリー4 add9
3dim7      # ディグリー3ディミニッシュセブンス
```

### オルタレーション（括弧なし・括弧あり両対応）
```
# 括弧なし表記（後方互換性）
1sus4      # ディグリー1サスフォー
4M7b5      # ディグリー4メジャーセブンスフラットファイブ

# 括弧あり表記（推奨）
1(sus4)    # ディグリー1サスフォー
4M7(b5)    # ディグリー4メジャーセブンスフラットファイブ
```

### テンション付き
```
1M7(b9)           # ディグリー1メジャーセブンスフラット9
5(b9)             # ディグリー5フラット9
2m7(b9,s11)       # ディグリー2マイナーセブンスフラット9シャープ11
4M7(9,s11)        # ディグリー4メジャーセブンス9シャープ11
1M7(b9,s11,b13)   # ディグリー1メジャーセブンスフラット9シャープ11フラット13
```

### オルタレーションとテンションの混在
```
# 括弧内にすべて含める（推奨）
1M7(b5,b9)        # ディグリー1メジャーセブンスフラットファイブフラット9
2m7(sus4,9)       # ディグリー2マイナーセブンスサスフォー9
4(b5,s11)         # ディグリー4フラットファイブシャープ11
1M7(sus4,b9,13)   # ディグリー1メジャーセブンスサスフォーフラット9サーティーンス

# 括弧外と括弧内の混在（両方とも保持される）
1M7b5(b9)         # 括弧外にb5、括弧内にb9（両方ともASTに保持される）
2m7sus4(9,s11)    # 括弧外にsus4、括弧内に9とs11
```

### ベース音付き
```
5/3              # ディグリー5分の3（ベース音3）
b7/5             # フラット7分の5（ベース音5）
1M7(b9)/3        # ディグリー1メジャーセブンスフラット9分の3
4M7(b5,b9)/b3    # ディグリー4メジャーセブンスフラットファイブフラット9分のフラット3
1/s4             # ディグリー1分のシャープ4
```

### 色アノテーション付き
```
1@red                # ディグリー1を赤で強調
3m7@blue             # ディグリー3マイナーセブンスを青で強調
s4M7(b5,b9)/b3@green # 複雑なコードにも末尾に付けられる
```

### 複雑な例
```
s4M7b5(b9,s11)/b3     # 括弧外b5 + 括弧内テンション（混在表記、両方保持）
s4M7(b5,b9,s11)/b3    # すべて括弧内表記（推奨）
1(sus4,b9,s11,b13)    # 複数オプションの組み合わせ
```

## 完全な入力例
```
1 3m7 4M7 5 | b7 1
s4M7(b5,b9,s11) 5 |
2m7(b5) 5(b9) 1M7(sus4,9)
```

### パース結果
```
Line 1: [Chord(1), Chord(3m7), Chord(4M7), Chord(5), BarLine, Chord(b7), Chord(1)]
Line 2: [Chord(s4M7(b5,b9,s11)), Chord(5), BarLine]
Line 3: [Chord(2m7(b5)), Chord(5(b9)), Chord(1M7(sus4,9))]
```

## AST定義

### TypeScript型定義
```typescript
type ParseResult = {
  score: ScoreAST          // 部分 AST（エラー行は除外される）
  errors: ParseError[]     // 行番号・位置つきの構造化エラー
}

type ScoreAST = {
  meta?: ScoreMeta         // フロントマター（宣言時のみ）
  sections?: SectionLabel[] // セクションラベル（行番号順）
  lines: Line[]
}

type SectionLabel = {
  line: number             // 元ソースの行番号（1-indexed）
  name: string
}

type ScoreMeta = {
  key?: string             // "C" | "G" | "F#" | "Bb" など
}

type Line = {
  line: number             // 元ソースの行番号（1-indexed）
  tokens: Token[]
  lyrics?: (string | null)[]  // コード系トークンに左から割り当てた歌詞断片
}

type ParseError = {
  kind: string             // "InvalidDegree" など
  message: string
  line?: number            // 1-indexed
  position?: number        // 行内の 0-indexed 文字位置
  token?: string
}

type Token = 
  | { type: 'Chord', chord: ChordNode }
  | { type: 'ChordGroup', chords: ChordNode[] }  // - 連結（1 スロット共有）
  | { type: 'BarLine' }
  | { type: 'Repeat' }                            // %（直前のコードの反復）
  | { type: 'NoChord' }                           // NC / N.C.
  | { type: 'EmptyBeat' }                         // _（空の拍）

type ChordNode = {
  accidental?: 's' | 'b'
  degree: '1' | '2' | '3' | '4' | '5' | '6' | '7'
  quality?: 'm' | 'dim' | 'aug'
  seventh?: Seventh
  alteration?: Alteration    // 括弧外のオルタレーション（後方互換性）
  options?: ChordOption[]    // 括弧内のオプション（推奨）
  bass?: BassNote
  color?: ChordColor         // 表示用の強調色
}

type ChordColor = 'blue' | 'red' | 'green'

type Seventh = 
  | { type: 'seventh', value: '7' | 'M7' | '6' }
  | { type: 'add', degree: '2' | '4' | '6' | '9' | '11' | '13' }

type Alteration = 
  | { type: 'b5' }
  | { type: 'sus', degree: '2' | '4' | '6' }

type ChordOption = Tension | Alteration

type Tension = 
  | { type: 'b9' }
  | { type: '9' }
  | { type: 's9' }
  | { type: '11' }
  | { type: 's11' }
  | { type: 'b13' }
  | { type: '13' }

type BassNote = {
  accidental?: 's' | 'b'
  degree: '1' | '2' | '3' | '4' | '5' | '6' | '7'  // 必須
}
```

### AST構築の注意点

**括弧外と括弧内のオルタレーションの混在**

入力: `1M7b5(b9)`

AST出力:
```typescript
{
  degree: '1',
  seventh: { type: 'seventh', value: 'M7' },
  alteration: { type: 'b5' },      // 括弧外のb5
  options: [{ type: 'b9' }]        // 括弧内のb9
}
```

両方のフィールドが同時に存在する可能性があることに注意してください。

## 正規表現パターン

コード全体をマッチする正規表現（疑似コード）：
```regex
^([sb])?                              # アクシデンタル
([1-7])                               # ディグリー
(m|dim|aug)?                          # クオリティ
(7|M7|6|add(?:2|4|6|9|11|13))?       # セブンス（7/M7/6が優先）
(b5|sus[246])?                       # オルタレーション（括弧外、後方互換性）
(?:\(((?:b9|s9|9|s11|11|b13|13|b5|sus[246])(?:, ?(?:b9|s9|9|s11|11|b13|13|b5|sus[246]))*)\))?  # 括弧内オプション（空白許容）
(?:/([sb])?([1-7]))?                 # スラッシュベース（ディグリー必須）
(?:@(blue|red|green))?               # 色アノテーション（末尾のみ）
$
```

## パーサー実装ガイドライン

### 字句解析 (Lexer)

1. 入力を行に分割し（行番号を保持）、行ごとに走査する
2. `#` 以降は行末までコメントとして無視する
3. 空白文字（スペース、タブ、連続する空白）、`|` でトークンを分割する
4. **括弧の内側では空白で分割しない**（括弧深度を追跡する）
5. 各トークンは行内の開始位置（0-indexed）を保持する（エラーの position 算出に使う）

#### 空白文字の扱い
- 単一スペース、複数スペース、タブ文字をすべて区切り文字として扱う
- 連続する空白文字は1つの区切りとして扱う
- 括弧 `(` `)` の内側の空白はトークンの一部として保持し、オプション解析時にトリムする

### 構文解析 (Parser)

1. コードトークンを正規表現または状態機械で解析
2. 各構成要素を**以下の順序で**順次抽出：
   - アクシデンタル（オプショナル）
   - ディグリー（必須）
   - クオリティ（オプショナル）
   - セブンス（オプショナル、**7/M7/6を優先的にマッチ**）
   - セブンスの重複チェック（セブンスを読んだ直後に再度セブンスがマッチする場合は DuplicateSeventh エラー。`1M77` / `17M7` / `1M7add9` など）
   - オルタレーション（括弧外、オプショナル）
   - 括弧内オプション（オプショナル、カンマで分割）
   - スラッシュベース（オプショナル、ただしスラッシュがあればディグリーは必須）
   - 色アノテーション（オプショナル、`@`があれば色名 blue/red/green は必須。トークン末尾のみ）
3. 括弧内オプションをパース：
   - `b5` → Alteration
   - `sus` + ディグリー → Alteration
   - その他 → Tension
4. ASTノードを構築

### セブンスのパース優先順位

セブンスのパースでは以下の順序で試行します：

1. `M7` （2文字なので7より先にチェック）
2. `7`
3. `6`
4. `add2`, `add4`, `add6`, `add9`, `add11`, `add13`

例：
- `1M7` → `{ type: 'seventh', value: 'M7' }`
- `17` → `{ type: 'seventh', value: '7' }`
- `1add9` → `{ type: 'add', degree: '9' }`

### 括弧内オプションのパース（疑似コード）
```
function parseOptions(content: string): ChordOption[] {
  // 空の括弧チェック
  if content.trim().isEmpty():
    throw Error("空の括弧は許可されていません")
  
  options = []
  parts = content.split(',')
  
  // カンマの不正使用チェック
  for (i, part) in enumerate(parts):
    if part.trim().isEmpty():
      throw Error("カンマの不正使用（先頭・末尾・連続）")
  
  for part in parts:
    trimmed = part.trim()
    
    if trimmed == "b5":
      options.push({ type: "b5" })
    else if trimmed.startsWith("sus"):
      degree = trimmed[3..]
      if degree not in ["2", "4", "6"]:
        throw Error("不正なsusディグリー")
      options.push({ type: "sus", degree })
    else:
      // Tensionとしてパース
      options.push(parseTension(trimmed))
  
  return options
}
```

### エラーハンドリング

パーサーは fail-fast せず、**行単位でエラーを収集**します。エラーのある行は部分 AST から除外され、`errors` 配列に `kind` / `message` / `line` / `position` / `token` を持つ構造化エラーとして記録されます。1 行に複数の不正トークンがあれば、その行から複数のエラーが報告されます。

以下の場合はエラーを返します：

#### 構文エラー
- 不正なディグリー（1-7以外）
- 不正なセブンス表記
- 不正なテンション表記（括弧内の認識できない要素は InvalidTension として報告される）
- 括弧の不一致
- 空の括弧 `()`
- カンマの不正使用（`(,b9)`, `(b9,)`, `(b9,,s11)`）
- 不正な色指定（`1@yellow`, `1@` など blue/red/green 以外。`1@red/3` のように色の後に構成要素が続く場合もエラー）
- 認識できない文字列

#### 意味エラー
- セブンスの重複（`1M7M7`, `17M7` など）
- スラッシュベースでディグリーが欠落（`1/`, `1/b` など。`1/8` のように 1-7 以外の度数が書かれている場合は構文エラー InvalidDegree になる）
  - なお `/3` のように**ルートの**ディグリーが欠落した入力は、構文エラー InvalidDegree（コード先頭のディグリー必須違反）に分類される
- 不正なsusディグリー（`sus1`, `sus5`, `sus7` など、2/4/6以外。括弧内の `sus11` / `sus`（度数なし）も同様）

#### フロントマターのエラー（v1.2）
- 閉じ `---` の欠落（UnclosedFrontmatter）
- 未知のフィールド（InvalidFrontmatterField。v1.2 は `key` のみ）
- 不正なキー名（InvalidKey。`A`-`G` + 任意の `#`/`b` 以外）

#### エラーメッセージ例
```
Error: Invalid degree '8' in chord '8M7'
Error: Empty parentheses are not allowed in '1M7()'
Error: Duplicate seventh specification in '1M77'
Error: Bass degree is required when slash is present in '1M7/'
Error: Invalid comma usage in '1M7(,b9)'
Error: Invalid color 'yellow' (must be blue, red, or green) in '1@yellow'
```

## 記法ガイドライン

### 推奨される記法

- **括弧内表記を優先**: `1M7(b5,b9)` ← 推奨
- **括弧外表記は最小限に**: `1M7b5(b9)` ← 許容されるが、統一性の観点から推奨しない

### 一貫性のある記述

同一楽譜内では、括弧の使い方を統一することを推奨します：
```
# 良い例（統一されている）
1M7(b5) 2m7(b9) 3(sus4,9)

# 避けるべき例（混在している）
1M7b5 2m7(b9) 3sus4(9)
```

### 空白文字について

- スペース、タブ、複数の連続する空白はすべて正常に処理されます
- ただし、可読性のため単一スペースの使用を推奨します

## Chord ブロックとフロントマター（v1.2）

> **実装状況（2026-07-03 更新）**: フロントマターのパースは Phase 3 で、タブ・キープルダウン表示と実音変換（移調）は Phase 4 で**実装済み**。ウィジェット HTML は `render_widget_html`、キー変更時の再描画は `parse_to_notes_html` が提供する（SSR とライブプレビューが同一の単一ソースを使う）。

### 目的

本 DSL は**ディグリー（度数）表記**である。読者・演奏者が実際のコード名で読めるように、ブロックにデフォルトの調（キー）を宣言でき、レンダリング後の表示を**ディグリー ⇄ コード名**のタブで切り替えられるようにする。さらにフロント側でキーを任意に変更（移調）できることを理想とする。

### フロントマター構文

ブロックの**先頭にのみ**、YAML 風のフロントマターを置ける：

```
:::
---
key: G
---
1 3m7 4M7 5 | b7 1
:::
```

```
フロントマター = "---" 改行 フィールド行* "---" 改行
フィールド行   = フィールド名 ":" 空白* 値
```

- フロントマターは**省略可能**。置く場合は 1 行目が `---` であること
- **フロントマターとコード譜本体の間は、閉じの `---` 行で区切る（必須）**。閉じ `---` が無い場合はエラー（UnclosedFrontmatter）
- **フロントマターを省略した場合（または `key` フィールドが無い場合）、キーは `C` をデフォルトとして扱う**。AST 上は宣言の有無を保持し（`meta.key` は宣言時のみ設定）、既定値 C の適用は表示層が行う
- v1.2 で定義するフィールドは **`key` のみ**
- キー名の書式: `A`〜`G` + 任意の `#` / `b`（例: `C`, `G`, `F#`, `Bb`）。**v1.2 ではメジャーキーのみ**（マイナーキーは未決事項 D10）
- エラー: 閉じ `---` の欠落（UnclosedFrontmatter）、未知のフィールド（InvalidFrontmatterField）、不正なキー名（InvalidKey）

### AST の拡張

```typescript
type ScoreAST = {
  meta?: ScoreMeta
  lines: Line[]
}

type ScoreMeta = {
  key?: string   // "C" | "G" | "F#" | "Bb" など
}
```

JSON 出力:

```json
{ "type": "ScoreBlock", "meta": { "key": "G" }, "lines": [ ... ] }
```

### 表示仕様（レンダリング後）— 実装済み

1. **タブ切り替え**: ブロックは「ディグリー」「コード」の 2 タブを持ち、読者が切り替えられる。**デフォルトタブはディグリー**（D11 決定）。タブはブラウザのタブのように下のパネルと一体に見えるデザイン（アクティブタブが区切り線と連結する）
2. **ディグリータブ**: ディグリーを**ローマ数字**で、アクシデンタルを **`♭` / `#` 記号**で表示する（後述「ディグリー表示の変換規則」）
3. **コードタブ**: キーに基づいてディグリーを実音コード名に変換して表示する。例（`key: G`）: `1 → G`、`3m7 → Bm7`、`b7 → F`、`s4M7(b5) → C#M7(b5)`、`5/3 → D/B`
4. **キープルダウン**: コードタブにキー選択プルダウンを置き、読者が**任意のキー**へ変更（移調）できる。初期値はフロントマターの `key`（未宣言なら既定値 `C`）。プルダウンのキー一覧は 12 半音（`C, D♭, D, E♭, E, F, F#, G, A♭, A, B♭, B`。宣言キーが一覧外の場合は追加される。D12 の暫定決定）
5. **色アノテーション**（`@red` 等）は両タブ・全キーで保持される
6. **コードスロットグリッド**（v1.2.5、v1.3 で改良）: 各コード・グループ・反復記号を**スロット**（グリッドの 1 マス）に置き、列数はスコア内の最大スロット数に揃える。行が違っても同じ位置のコード同士が縦に揃う（実譜面のシステム整列のイメージ）。小節線 `|` はスロットを消費せず、**スロット境界の縦線**として描画される。行頭・行末の `|` はそれぞれ開始・終止の縦線になる。列幅は内容に応じて広がる（`minmax(max-content, 1fr)`）ため、**長いコードが `...` に切り詰められることはない**。`-` グループは 1 スロットを**等間隔で分け合う**（2 連結ならスロットの頭と中間 — 拍位置のイメージ）
7. **キープルダウンの異名同音**（v1.2.5）: フロントマターで宣言したキーが一覧のキーと異名同音の場合（例: `C#` と `Db`）、一覧のそのスロットを宣言の綴りで置き換える（末尾に追加しない）

### ディグリー表示の変換規則（v1.2.1 — 実装済み）

DSL の入力は `1`〜`7` / `b` / `s` のまま変えず、**表示層**で以下に変換する：

| 入力 | 表示 |
|------|------|
| `1`〜`7`（ディグリー） | `I`〜`VII`（ローマ数字・大文字） |
| `b`（フラット） | `♭` |
| `s`（シャープ） | `#` |
| ベース音のディグリー | 同様にローマ数字（`5/3 → V/III`） |
| 括弧内テンション | `(b9,s11) → (♭9,#11)` |

例: `s4M7(b5,b9,s11)/b3` → `#IVM7(♭5,♭9,#11)/♭III`、`3m7` → `IIIm7`

クオリティ（m/dim/aug）・セブンス・テンションの表示位置・書体の調整（リードシート風の上付き表示等）は Phase 4 で行う。

### 変換規則 — 実装済み

- ディグリー＋アクシデンタルを、キーのメジャースケール上の音名にマップする（`b7` = キーの短 7 度上、`s4` = 増 4 度上）
- クオリティ・セブンス・オルタレーション・テンション・括弧内オプションは表記をそのまま引き継ぐ（`3m7(b5)` → `Bm7(b5)` @ G）
- スラッシュベースも同じ規則で変換する
- スペリングはメジャースケールの**レター（C-B）単位**で解決するため異名同音が正しくなる（例: key F# の `7` は `E#`、key Bb の `b3` は `D♭`）。DSL のアクシデンタル（b/s）は同じレターの半音変化として適用する（key C の `b7` は `B♭`）。参照実装: tonal.js の `Progression.fromRomanNumerals`、ChordSheetJS の `toChordSymbol(key)`
- 変換は **AST → 表示の純関数**とし、パース結果（ディグリー AST）自体は変更しない

### Markdown への埋め込み（v1.2.1 — 実装済み）

ブログ（markdown.mbt フォーク）の Markdown 文書には、**`:::` 単独のフェンス**でコード譜ブロックを埋め込む：

```
:::
1 3m7 4M7 5 | b7 1
:::
```

- 開き・閉じとも `:::`（コロン 3 つ以上）の行。言語名は不要（このプラットフォーム特有のコード譜専用ブロック）
- 互換として ` ```chord `（フェンスドコードブロック + 言語タグ）も引き続き動作する
- ` ```chord:code ` と書くと譜面ではなく DSL ソースのハイライト表示になる
- 将来 `:::名前` 形式を別種のコンテナに使う可能性は予約（現状は `:::` 直後の文字列は無視され、常にコード譜ブロックとして扱われる）
