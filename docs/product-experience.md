# プロダクト体験仕様書

最終目標のブログプラットフォームにおける体験を、3 つのロール(筆者・読者・演奏者)のジャーニーとして固定する。以降の各開発フェーズの完了条件はこの文書を参照する([roadmap.md](./roadmap.md) 参照)。

作成日: 2026-07-03 / 最終更新: 2026-07-04(仕様 v1.0 確定に合わせて改訂)

## プロダクトの一文定義

**Markdown で書けるブログに `:::` コード譜ブロックを足し、数字ディグリー記法のコード譜を書ける・読める・聴ける・演奏に使えるようにするプラットフォーム。**

---

## ジャーニー 1: 筆者(ブログを書く人 = プロジェクトオーナー自身)

1. **執筆**: Markdown エディタで記事を書く。コード進行の説明をしたい箇所で `:::` ブロックを開き、DSL(`1 3m7 4M7 5 | b7 1` など)を打つ
2. **ライブプレビュー**: 打鍵ごとにプレビューペインのコード譜が更新される(JotChord 型の体験)。chord ブロック以外の Markdown 編集ではコード譜は再描画されない
3. **強調**: 文脈上目立たせたいコード(記事の主題となる転回・借用和音など)に `@red` / `@blue` / `@green` を付けると、プレビューと公開ページの両方で色付き表示される
4. **エラー修正**: 打ち間違い(`8m7`、`1M7()` など)をすると、プレビューが行番号・位置つきのエラー表示に変わる。ブロック全体やページは壊れない。修正すると即座に譜面表示へ戻る
5. **公開**: 記事を公開すると、プレビューと同一の見た目のコード譜が公開ページ(SSR/静的 HTML)に出る

### 筆者体験の品質条件

- エラーはブロック内で完結し、部分的に正しい行は描画され続ける(エラー耐性パーサ)
- プレビューと公開ページの描画結果が一致する(MoonBit 側レンダラの単一ソース性)
- エディタ内シンタックスハイライトは将来スコープ

## ジャーニー 2: 読者(記事を読む人)

1. **閲覧**: 公開 URL で記事を開くと、本文の Markdown とコード譜が一体でレンダリングされている
2. **視線誘導**: 筆者が色を付けたコードが強調表示され、本文の解説(「この `@red` の s4M7(b5) が肝で…」)と対応づけて読める
3. **環境**: ライト/ダークテーマの両方で読める。スマートフォンでは小節が適切に折り返される。印刷してもコード譜が崩れない

## ジャーニー 3: 演奏者(コード譜を使う人)

1. **タブ切替**: 記事の chord ブロックには「ディグリー」「コード」の 2 タブがあり、切り替えて読める。ディグリータブはローマ数字表示(`I IV V`)、コードタブは筆者がブロックのフロントマター(`key: G`)で宣言したキーで実音表示される(`1 4 5` → `G C D`)。key 未宣言のブロックは既定値 C で実音表示できる
2. **任意キーへの移調**: コードタブのプルダウンで別のキーを選ぶと、その場でそのキーのコード名に切り替わる(`A` を選べば `A D E`)。自分の楽器や歌いやすさに合わせて移調して使える
3. **再生と演奏利用**: ▶ ボタンで表示中のキーの和音を確認でき、実在曲 1 曲分のコード譜が画面または印刷で演奏に使える品質で表示される

## MVP スコープ

### 実装済み(仕様 v1.0)

- DSL 全構文のパースと HTML 表示(コード・グループ・%・N.C.・空拍・セクション・歌詞・色)
- `:::` ブロックのライブプレビューと SSR/静的 HTML の同一描画
- ディグリー ⇄ コードのタブ切替、キープルダウンによる任意移調
- 再生(キー・bpm・拍子に追従、ベースの拍刻み、再生カーソル)
- 行番号・位置つきエラーの行内埋め込み表示(正しい行は描画継続)
- 譜面の PNG 画像コピー

### 将来スコープ

- リピート記号・ボルタ・終止線(予約トークンのまま)
- マイナーキー(`key: Am`)対応
- 記事管理(下書き・タグ・一覧)の本格機能と公開基盤(ブログとしてのデプロイ)
- エディタ内シンタックスハイライト
- 色のカスタマイズ(任意色・テーマ変数)— 3 色で運用し、必要になってから拡張

## 色アノテーションの設計意図

- 用途は**記事の文脈上の強調**であり、音楽記号ではない。よって AST 上も `color` として他の音楽的構成要素から独立させる
- 色数を 3 色に固定するのは、記事内の視覚言語を単純に保つため(「赤 = 主題」「青 = 対比」など筆者が記事ごとに意味づけできる余地を残す)
- 表示はレンダラ側で CSS クラス(例: `chord--red`)に変換し、実際の色価はテーマ(ライト/ダーク)側で決める
