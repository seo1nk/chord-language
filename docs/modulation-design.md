## 1. 記法

**`[key: 値]` の単一構文。** 値は相対(半音数)と絶対(キー名)の両対応。

```
転調指示   = "[" 空白* "key" 空白* ":" 空白* 転調値 空白* "]"
転調値     = 相対値 | 絶対キー
相対値     = ("+" | "-") 整数            ; 半音数 1〜11(0 と 12 以上は InvalidKeyChange)
絶対キー   = ("A"〜"G") ("#" | "b")?     ; フロントマターの key と同一文法(メジャーのみ)
```

- **単独行**(セクションラベルと同格の第 4 の行種)と**コード行内のインライントークン**の両方に書ける。意味は同じ「この位置以降のディグリーは新キー基準」
- 判定は「trim 済み内容が `key` 空白* `:` で始まる」。**プレフィックスに一致したら、値が不正でも必ず InvalidKeyChange**(`[key : G]` も転調として拾う。タイポが黙ってセクションバッジ化する事故を防ぐ)
- `[Key: F]`(大文字)は従来どおりセクションラベル(小文字強制は色名・NC と同じ流儀)
- グループ(`-` 連結)の中では使えない(`1-[key:+2]5` は従来どおり不正。仕様に明記 + テスト追加)
- レキサ: **トークン先頭の `[` のみ**ブラケット開始(`1[key:+2]` は従来どおりエラー。既存エラー空間の書き換えを最小に)

### 実例

(a) ラスサビ転調(単独行):

```
[ラスサビ]
[key: +1]
| 4M7 5 | 3m7 6m7 | 2m7 5 | 1 % |
```

(b) B メロ 2 小節ごと全音上げ(インライン):

```
[Bメロ]
| 4 5 | 3m 6m | [key: +2] 4 5 | 3m 6m | [key: +2] 4 5 | 57 |
```

### 却下案(chord.md 設計判断メモに記録すること)

`[+2]` / `[F]` プレフィックスなし(セクション名 [A] と衝突)/ セクション属性化 `[ラスサビ +1]` / 新シグル `^+2` 等 / フロントマターに転調表 / `%%` 的な転調反復略記 / 行レベルのみ・インラインのみ。

## 2. 意味論

- 指示以降(文書順)のディグリーは新キー基準。相対は現在キーからの合成(`+1` の後の `+1` = 計 +2)。効果は次の指示まで
- **キープルダウン = 「初期キーの差し替え」に再定義**。`parse_to_notes_html(input, key)` / `parse_to_playback(input, key)` の key 引数 = 選択中の初期キー(シグネチャ不変)
- δ = (pc(選択キー) − pc(宣言キー)) mod 12 を**文中の絶対キー指定にも一様適用**。2 つの不変量として仕様化: 「pc(K_eff) ≡ pc(K_src) + δ」「転調幅の保存」(移調しても曲内の転調構造が変わらない)
- 内部状態は **KeyCursor = 現在キー(KeyInfo)+ δ** の文書順走査(「初期キーからのオフセットへ正規化」はマイナーキー拡張を塞ぐため不採用)
- **スペリング**: δ=0 の絶対指定は原文綴りを尊重。相対指定・δ≠0 は正準 12 キー表 `C,Db,D,Eb,E,F,F#,G,Ab,A,Bb,B` で綴る。**この表は transpose.mbt に単一ソース化し、widget.mbt の `widget_keys()` はそこから導出**(二重定義禁止)。実装時にプレイグラウンドで E/B/F# 起点の連続 +1/+2 を目視確認し、結果を設計判断メモに 1 度だけ記録
- **`%` / `_` / NC は実音固定**(転調をまたいでも直前の響き。playback の last_voicing と一致 = 実装差分ゼロ)。チートシートに「転調指示は次に書いたコードから効く(% はまたがない)」を明記
- 転調指示は**スロット・歌詞・時間・`.chord-cell` を消費しない** → `cursor.cell` 契約・PNG 書き出し・chord-widget.ts は無変更

## 3. AST / JSON(追加のみ)

```moonbit
pub enum KeyChange {
  RelativeKey(Int)     // -11〜-1, +1〜+11
  AbsoluteKey(String)  // "F" "Eb" など(検証済み)
}
// Token に KeyChangeTok(KeyChange) を追加。単独行は「そのトークン 1 つの Line」
```

- キー領域の解決は AST に持たせず、**消費側(transpose / playback / renderer)の文書順 fold** で行う(lines / sections / modulations の 3 ストリームマージを避ける)
- JSON: `{ type: "KeyChange", change: { type: "relative", semitones } | { type: "absolute", key } }`

## 4. 表示

- **ディグリータブにも転調チップを出す**(キー非依存タブなのでソース忠実形: `key +1` / `key F`。相対・絶対で同形式)
- コードタブは解決後: 単独行は `Key D♭ (+1)` 併記、インラインは幅優先でキー名のみ
- 視覚言語: **セクション = 塗り(白抜き反転)/ 転調 = アウトライン**(currentColor 枠 + 背景透過 → ライト/ダーク/PNG 白地すべて追加実装なしで成立)。CSS は `.chord-modline`(単独行・全幅)/ `.chord-mod`(セル付きチップ)
- インライン = 次のコードセルの前置チップ。**行末のインライン指示 = 直前セルの後置チップ**(段末の予告調号の慣習。行をまたぐ pending 状態をレンダラに持ち込まない)。スコア末尾の宙吊り指示も末尾バッジとして描画(書いたものを黙って消さない)
- `data-chord-modulated` 属性・転調有無での aria-label 分岐は**初期実装では見送り**(必要になったら additive に追加)

## 5. エラー処理

- 新設 kind は **`InvalidKeyChange` の 1 つだけ**(不正値 `[key: H]` `[key: +0]` `[key: +13]` `[key:]`、未クローズ `[key:+2` を包含)。行単位で埋め込み、正しい行は生き続ける(既存原則)
- `key:` で始まらないブラケットは**従来挙動を完全保存**(コード行中の `[サビ] 1 4` → InvalidDegree。誘導したければメッセージ増強のみで kind は増やさない)
- 転調単独行の直後の歌詞行: 機構上は TooManyLyricFragments のパスに落ちるが、**専用メッセージ**(「歌詞行はコード行の直後に書きます(転調指示行には付けられません)」)に分岐する
- **インライン転調を含む行が別エラーで行ごと除外されると、以降のキー領域は指示なしとして継続する** — 「エラー行は存在しないものとして残りを描画する」原則の帰結として chord.md エラーハンドリング節に明記 + spec-test-matrix にケース追加

## 6. 互換性・バージョニング(決定済み)

- **v1.1** として出す。同時に api.md §4 を改訂し、次の予約メカニズムを約束に組み込む:
  - 「`[小文字識別子: 値]` 形式のブラケット(単独行・インライントークンとも)は**ディレクティブ予約領域**とする。予約領域への意味付与は、実在文書への影響がないと判断した場合に限りマイナーバージョンで行える(改訂履歴に必ず明記)」
  - これにより将来の `[bpm: 140]` `[time: 3/4]`(途中変更)やモード対応も同じ枠で追加できる
- chord.md 改訂履歴に「`[key: …]` という名前のセクションだった文書は転調指示として再解釈される」ことを明記

## 8. 検証項目(実装時)

- (a)(b) 両パターンのゴールデン(実音・再生 MIDI・バッジ表示)
